# Checklist antes de publicar el portfolio

## Identidad profesional

- [ ] El titular describe el rol objetivo: junior backend / producto / full-stack con foco backend.
- [ ] El texto no usa claims exagerados como “experto”, “arquitecto” o “senior”.
- [ ] GitHub, LinkedIn, CV y portfolio cuentan la misma historia.

## Proyectos

- [ ] Aegis tiene demo o vídeo corto.
- [ ] SharedVault tiene demo o mockup funcional.
- [ ] BudgetWise tiene capturas reales.
- [ ] Cada proyecto tiene enlace a GitHub.
- [ ] Cada README explica problema, stack, instalación, arquitectura y roadmap.
- [ ] Las capturas no son decorativas: enseñan flujo, arquitectura o decisión técnica.

## GitHub

- [ ] Repositorios fijados: portfolio, Aegis, SharedVault, BudgetWise y máximo dos secundarios.
- [ ] Commits limpios con convención `feat:`, `fix:`, `docs:`, `refactor:`.
- [ ] Issues abiertas con mejoras reales.
- [ ] Releases `v0.1.0`, `v0.5.0` o `v1.0.0` en proyectos principales.
- [ ] `.env.example` en proyectos backend/cloud.

## Calidad técnica

- [ ] `npm run build` funciona.
- [ ] Diseño responsive comprobado en móvil.
- [ ] No hay enlaces rotos.
- [ ] Los botones de demo/GitHub ya no están en “pendiente”.
- [ ] Lighthouse revisado para rendimiento, accesibilidad y SEO.

## Entrevista

- [ ] Puedes explicar Aegis en 90 segundos.
- [ ] Puedes explicar por qué SharedVault usa AWS S3 y URLs firmadas.
- [ ] Puedes explicar cómo organizarías una v2 de cada proyecto.
- [ ] Puedes hablar de trade-offs sin sonar artificial.
